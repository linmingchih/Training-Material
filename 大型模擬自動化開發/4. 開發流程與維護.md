# 開發流程與維護

這份文件聚焦於大型模擬自動化系統的落地流程、維運機制與持續迭代策略。

## 0. 基礎環境安裝
在開始之前，請確保您的開發環境中已安裝以下核心工具。這些工具是支撐整個開發、管理與部署流程的基石。

### 安裝 uv, Git, 與 npm
- **uv**: 一個極速的 Python 套件安裝與管理工具。在 Windows 上，您可以透過確保已安裝 Python 和 pip，然後在命令提示字元或 PowerShell 中執行以下指令來安裝：
  ```cmd
  pip install uv
  ```
  如果尚未安裝 Python，建議從 [Python 官方網站](https://www.python.org/downloads/windows/) 下載安裝程式。

- **Git**: 業界標準的分散式版本控制系統，用於追蹤程式碼變更。
  - **建議安裝方式 (官方下載)**: 從 [Git 官方網站](https://git-scm.com/downloads) 下載 Windows 安裝程式，並依照指示完成安裝。
  - **透過套件管理器 (推薦)**: 如果您已安裝 [Winget](https://learn.microsoft.com/zh-tw/windows/package-manager/winget/) (Windows 封裝管理員) 或 [Chocolatey](https://chocolatey.org/)，可以在命令提示字元或 PowerShell 中執行以下指令：
    ```cmd
    winget install Git.Git
    # 或者如果您使用 Chocolatey
    # choco install git
    ```

- **npm**: Node.js 的套件管理器，主要用於管理前端專案的依賴。npm 會隨 Node.js 一併安裝。
  - **建議安裝方式 (官方下載)**: 從 [Node.js 官方網站](https://nodejs.org/zh-tw/download/current) 下載 Windows 安裝程式，並依照指示完成安裝。
  - **透過套件管理器 (推薦)**: 如果您已安裝 Winget 或 Chocolatey，可以在命令提示字元或 PowerShell 中執行以下指令：
    ```cmd
    winget install OpenJS.Nodejs
    # 或者如果您使用 Chocolatey
    # choco install nodejs
    ```

安裝完成後，您可以在命令提示字元或 PowerShell 中執行以下指令來確認安裝是否成功：
```cmd
uv --version
git --version
npm --version
```

## 1. 開發環境安裝與系統管理
在大型專案中，環境的一致性與可重現性是開發的基礎。

### Python 管理工具：uv
`uv` 是目前 Python 社群中最快的套件管理器，整合了環境管理與依賴解析。
- **專案初始化**: `uv init --python 3.10` 快速建立標準專案結構。
- **同步環境**: `uv sync` 確保團隊開發環境一致。
- **依賴管理**: `uv add <package>` 自動處理相依性。
  - 範例: `uv add fastapi uvicorn pyaedt[all]` 可同時安裝多個套件及其所有依賴。
- **虛擬環境管理**: 自動建立 `.venv`，避免系統路徑衝突。

## 2. 前端框架
現代前端開發通常依賴於高效的框架來加速開發流程與提高應用效能。本專案推薦使用 Vite 搭配適當的框架（如 React, Vue 或 Svelte）。

以下是 Vite 專案的常用命令：
- `npm create vite@latest`: 初始化一個新的 Vite 專案。
- `npm run dev`: 啟動開發伺服器，提供即時重載功能。
- `npm run build`: 將專案打包成生產環境可部署的靜態檔案。
- `npm run preview`: 在本地預覽生產環境的打包結果。

## 3. 版本控制系統 (Version Control)
程式碼與模擬流程的迭代需要嚴格的管理。

### Git / GitHub Desktop
- **程式碼追蹤**: 使用 `git init` 開啟版本監控。
- **分支管理 (Branching)**: 區分開發版 (dev) 與穩定版 (main)。
- **協作開發**: 透過 Pull Request 進行代碼審查，確保腳本品質。
- **常用 Git 指令**:
  - `git status`: 檢查工作區與暫存區的狀態。
  - `git add <file>`: 將檔案加入暫存區。
  - `git commit -m "Message"`: 將暫存區的變更提交到本地儲存庫。
  - `git push`: 將本地儲存庫的變更推送到遠端儲存庫。
  - `git pull`: 從遠端儲存庫拉取最新變更並合併到本地。
- **與模擬檔案的配合**: 哪些檔案該進 Git (.py, .ts)，哪些該 ignore (.aedt, .results)。

## 4. 自動化部署與環境維護
大型專案的分發必須「一鍵啟動」，降低環境設定門檻。

### 啟動腳本核心邏輯 (.bat)
一個健壯的啟動腳本應包含以下邏輯：
1. **環境檢測**: 確保系統安裝 `uv`。
2. **依賴同步**: 執行 `uv sync`，自動建立或更新 `.venv`。
3. **並行啟動**: 背景執行 FastAPI 伺服器並同步開啟 `pywebview` 視窗。

```batch
@echo off
:: 1. 環境同步
call uv sync
if %errorlevel% neq 0 (
    echo [ERROR] Environment sync failed.
    pause
    exit /b
)

:: 2. 啟動後端與 UI
echo [INFO] Starting Simulation System...
start /b uv run uvicorn backend.main:app --host 127.0.0.1 --port 8000
uv run python run_app.py
```

## 5. AI 驅動的開發系統 (LLM + Agents)
利用大模型加速開發，並讓系統具備一定的「工程推理」能力。

### Coding Agent 與 Skill 系統
- **SKILL 載入**: 建立自定義的系統提示詞 (System Prompt)，將 PyAEDT/PyEDB 的專業知識 (Documentation) 注入 LLM。
- **Context Injection**: 讓 AI 讀取當前的 EDB 結構或報表資料，自動生成修改建議腳本。
- **Human-in-the-loop**: 關鍵步驟（如刪除重要 Net 或修改層疊）由 AI 生成建議，經工程師按鈕確認後才執行。

### 範例驅動開發 (Few-shot)
- **高品質範例庫**: 維護一個標準化的 `scripts/examples` 資料夾作為 AI 的學習參考 (RAG)。
- **自動修復流程**: 當腳本執行報錯時，將 Traceback 丟回 AI 進行原因分析、代碼修復與重新提交驗證。

## 6. 軟體品質與自動化測試
確保開發進度加快的同時，不會破壞原有功能。

### 數據驗證與 Pydantic
- **輸入前端檢查**: 避免用戶輸入非法字符或超出範圍的參數（如負數頻率）。
- **Schema 驗證**: 使用 Pydantic 模型定義 API 接口，確保前後端通訊協議一致。

### 單元測試 (Unit Testing)
- **邏輯抽離**: 將模擬邏輯（PyAEDT）與伺服器邏輯（FastAPI）分離，方便針對單一功能編寫測試腳本。
- pyaedt/pyedb腳本以參數方式傳遞，並另寫測試腳本。

## 7. 數據視覺化與工程報表策略
將模擬結果轉化為「可決策」的資訊。

### 交互式視圖 (Interactive View)
- **多維度對比**: 讓用戶能在同一張 Plotly 圖表中對比不同版本 (Versions) 或不同 Case 的 S 參數。
- **過渡帶分析**: 自動標註並判定 Pass/Fail（如 Mask 檢查）。

### 自動化報表產出
- **PDF/HTML 導出**: 結合 Jinja2 模板與 Matplotlib，一鍵生成正式的工程報表。
- **數位孿生預覽**: 在介面上直接渲染 EDA 工具生成的 3D 結構圖，方便確認物理模型的正確性。

## 8. 說明書與知識傳承
- **API 文件**: 自動根據 Docstring 生成（如 Sphinx），方便後續維護與二次開發。
- **使用手冊**: 專為 UI 使用者撰寫的圖文點擊手冊與功能說明。
- **範例庫與教材**: 提供標準的範例專案，作為 AI 學習或新人培訓的基準。

## 9. 持續更新與迭代
一個成功的自動化系統並非一蹴可幾，而是需要透過不斷的迭代與優化來持續進化。這個過程仰賴於一個結構化的反饋循環與敏捷的開發文化。

### 敏捷開發與反饋循環
- **小步快跑**: 採用敏捷開發（Agile）原則，將大型功能拆解為小而可交付的任務。每次迭代（Sprint）都專注於解決一到兩個核心問題或增加一個實用功能。
- **用戶反饋機制**: 在系統中建立一個簡單的「意見反饋」或「問題回報」按鈕。讓終端使用者能輕鬆地將遇到的問題、操作不便之處或新功能建議直接提交給開發團隊。
- **定期審查會議**: 定期（例如每兩週）與核心使用者開會，展示最新的開發進度，並收集他們對系統的直接反饋。

### 功能需求的優先級排序
- **影響力 vs. 成本分析**: 開發資源永遠是有限的。利用一個簡單的四象限矩陣（高影響力/低成本、高影響力/高成本等）來評估每個待辦事項，優先處理那些能以最小開發成本帶來最大效益的功能。
- **錯誤修復優先**: 系統的穩定性是使用者信任的基礎。通常，修復已知的錯誤（Bug）應具有比開發新功能更高的優先級。

## 結語
從單純的腳本撰寫到開發大型模擬自動化系統，關鍵在於把架構設計與開發維運拆分治理。以清晰架構支撐功能擴展，以穩定流程支撐長期演進，才能建立真正可持續的工程資產流水線。
