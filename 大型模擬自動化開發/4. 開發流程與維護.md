# 開發流程與維護

這份文件聚焦於大型模擬自動化系統的落地流程、維運機制與持續迭代策略。
## 1. 使用者故事

使用者故事是一種以使用者角度描述需求的方法，重點不是系統內部怎麼實作，而是使用者在什麼情境下、希望完成什麼事情、以及最後要得到什麼結果。對模擬自動化系統來說，使用者故事可以幫助我們先釐清真實工作流程，例如上傳電路板檔案、選擇要分析的 nets、設定模擬頻率、執行分析並輸出報告，讓開發目標更貼近實際使用情境，也更容易對應後續的功能設計、介面流程與驗收標準。




## 2. 分析需求，資源及架構

在規劃模擬自動化系統時，除了功能本身，還必須同步考慮授權條件、作業瓶頸、硬體資源、使用模式與資料流向，才能讓系統真的能穩定落地。

### License 與授權條件
- 確認模擬工具、求解器與相關外掛是否需要獨立 License，避免腳本已完成但無法實際啟動。
- 評估授權是否支援多使用者併發、遠端執行與排程佇列，避免因 License 數量不足造成等待。
- 若系統包含第三方前端套件、報表元件或 AI 服務，也需先確認其商用或內部使用授權限制。

### 作業瓶頸
- 模擬求解時間通常是主要瓶頸，特別是在高頻掃描、長時域分析或大型電路板模型中。
- 資料前處理與結果整理也可能成為瓶頸，例如檔案解析、net 篩選、圖表生成與報告輸出。
- 若採遠端執行，網路傳輸與檔案同步速度也會影響整體體驗，尤其是大型結果檔與圖片輸出。

### 電腦規格與執行資源
- 近端執行時，建議確認 CPU 核心數、記憶體容量、硬碟讀寫速度與 GPU 是否符合模擬需求。
- 若模型較大或結果較多，應優先考慮足夠的 RAM 與 SSD，避免因交換記憶體導致明顯降速。
- 遠端執行時，伺服器端應配置較高規格的運算資源，使用者端則以穩定的介面與檔案傳輸能力為主。

### 使用方式：近端與遠端
- 近端模式適合單人開發、除錯與快速驗證，優點是回應快、流程簡單、資料不必頻繁傳輸。
- 遠端模式適合正式分析、多人共用與高資源需求任務，可將耗時運算放在伺服器或工作站上執行。
- 若系統同時支援兩者，應明確區分本機執行與遠端提交的入口，避免使用者混淆操作路徑。

### 輸入與輸出
- 輸入可包含電路板檔案、net 清單、模擬參數、頻率範圍、材料設定與分析條件。
- 輸出可包含模擬結果、波形圖、S 參數、Pass/Fail 判定、報告檔與可追溯的日誌紀錄。
- 介面應清楚顯示哪些輸入是必要項、哪些輸出會自動產生，讓使用者能在執行前先確認內容是否完整。

### 同步與非同步
- 同步模式適合短時間任務，使用者送出後直接等待結果，流程直覺但會佔用操作時間。
- 非同步模式適合長時間模擬，使用者送出任務後可先離開，待系統完成後再回來查看結果或通知。
- 若任務耗時不可預期，建議以工作佇列、狀態查詢與結果通知的方式管理，讓前端能顯示執行中、完成、失敗等狀態。




### 架構建議

- 前端負責檔案上傳、參數設定與結果展示。
- 後端負責參數驗證、任務排程、模擬呼叫與報告彙整。
- 模擬引擎或外部工具應以獨立模組包裝，方便日後替換、測試與維護。

## 3. 基礎環境與工具

在開始之前，請確保您的開發環境中已安裝以下核心工具。這些工具是支撐整個開發、管理與部署流程的基石。

### 3.1 安裝 uv、Git 與 npm

- **uv**：一個極速的 Python 套件安裝與管理工具。在 Windows 上，您可以先確認已安裝 Python 與 pip，再執行下列指令安裝：

	```cmd
	pip install uv
	```

	如果尚未安裝 Python，建議先從 [Python 官方網站](https://www.python.org/downloads/windows/) 下載安裝程式。

- **Git**：業界標準的分散式版本控制系統，用於追蹤程式碼變更。

	- 建議安裝方式：從 [Git 官方網站](https://git-scm.com/downloads) 下載 Windows 安裝程式並依指示完成安裝。
	- 若已安裝 Winget 或 Chocolatey，也可以直接使用套件管理器：

		```cmd
		winget install Git.Git
		# 或者如果您使用 Chocolatey
		# choco install git
		```

- **npm**：Node.js 的套件管理器，主要用於管理前端專案的依賴。npm 會隨 Node.js 一併安裝。

	- 建議安裝方式：從 [Node.js 官方網站](https://nodejs.org/zh-tw/download/current) 下載 Windows 安裝程式並依指示完成安裝。
	- 若已安裝 Winget 或 Chocolatey，也可以直接使用套件管理器：

		```cmd
		winget install OpenJS.Nodejs
		# 或者如果您使用 Chocolatey
		# choco install nodejs
		```

安裝完成後，可以在命令提示字元或 PowerShell 中執行以下指令確認安裝是否成功：

```cmd
uv --version
git --version
npm --version
```

### 3.2 Python 管理工具：uv

`uv` 是目前 Python 社群中速度非常快的套件管理器，整合了環境管理與依賴解析。

- 專案初始化：`uv init --python 3.10` 可快速建立標準專案結構。
- 同步環境：`uv sync` 可確保團隊開發環境一致。
- 依賴管理：`uv add <package>` 可自動處理相依性。
- 範例：`uv add fastapi uvicorn pyaedt[all]` 可同時安裝多個套件及其依賴。
- 虛擬環境管理：自動建立 `.venv`，避免系統路徑衝突。

## 4. 開發管理與版本控制

程式碼與模擬流程的迭代需要嚴格的管理，否則很容易出現腳本散落、版本不一致或無法回溯的問題。

### 4.1 前端框架

現代前端開發通常依賴於高效的框架來加速開發流程與提高應用效能。本專案可使用 Vite 搭配適當框架（如 React、Vue 或 Svelte）。

- `npm create vite@latest`：初始化新的 Vite 專案。
- `npm run dev`：啟動開發伺服器，提供即時重載功能。
- `npm run build`：將專案打包成生產環境可部署的靜態檔案。
- `npm run preview`：在本地預覽生產環境的打包結果。

### 4.2 Git / GitHub Desktop

- 程式碼追蹤：使用 `git init` 開啟版本監控。
- 分支管理：區分開發版（dev）與穩定版（main）。
- 協作開發：透過 Pull Request 進行代碼審查，確保腳本品質。
- 常用 Git 指令：

	- `git status`：檢查工作區與暫存區的狀態。
	- `git add <file>`：將檔案加入暫存區。
	- `git commit -m "Message"`：將暫存區的變更提交到本地儲存庫。
	- `git push`：將本地儲存庫的變更推送到遠端儲存庫。
	- `git pull`：從遠端儲存庫拉取最新變更並合併到本地。

- 與模擬檔案的配合：清楚區分哪些檔案該進 Git（例如 `.py`、`.ts`），哪些該 ignore（例如 `.aedt`、`.results`）。

## 5. 自動化部署與環境維護

大型專案的分發必須「一鍵啟動」，降低環境設定門檻，同時保留必要的停止、重置與故障排除機制。

### 5.1 啟動腳本核心邏輯（.bat）

一個健壯的啟動腳本應包含以下邏輯：

1. 環境檢測：確認系統已安裝 `uv`。
2. 依賴同步：執行 `uv sync`，自動建立或更新 `.venv`。
3. 並行啟動：背景執行 FastAPI 伺服器並同步開啟 `pywebview` 視窗。

```batch
@echo off
:: 1. 環境同步
call uv sync
if %errorlevel% neq 0 (
		echo [ERROR] Environment sync failed.
		pause
		exit /b
)

:: 2. 啟動後端與 UI
echo [INFO] Starting Simulation System...
start /b uv run uvicorn backend.main:app --host 127.0.0.1 --port 8000
uv run python run_app.py
```

### 5.2 停止與重置腳本

除了啟動腳本之外，也需要提供對應的停止與重置工具，方便使用者在切換案例、重新整理環境或排除異常時快速操作。`stop.bat` 通常用來結束後端服務、關閉相關程序並釋放執行中的資源；`reset.bat` 則用來清理暫存資料、重建環境或回復到乾淨狀態，避免前一次執行留下的檔案或設定影響下一次分析。這兩個腳本和啟動腳本搭配使用，才能形成完整的維護流程。

## 6. AI 驅動的開發系統（LLM + Agents）

利用大模型加速開發，並讓系統具備一定的工程推理能力。

### 6.1 Coding Agent 與 Skill 系統

- SKILL 載入：建立自定義的系統提示詞（System Prompt），將 PyAEDT / PyEDB 的專業知識注入 LLM。
- Context Injection：讓 AI 讀取當前的 EDB 結構或報表資料，自動生成修改建議腳本。
- Human-in-the-loop：關鍵步驟（如刪除重要 Net 或修改層疊）由 AI 生成建議，經工程師按鈕確認後才執行。

### 6.2 範例驅動開發（Few-shot）

- 高品質範例庫：維護一個標準化的 `scripts/examples` 資料夾作為 AI 的學習參考（RAG）。
- 自動修復流程：當腳本執行報錯時，將 Traceback 丟回 AI 進行原因分析、代碼修復與重新提交驗證。

## 7. 軟體品質與自動化測試

確保開發進度加快的同時，不會破壞原有功能。

### 7.1 數據驗證與 Pydantic

- 輸入前端檢查：避免用戶輸入非法字符或超出範圍的參數（如負數頻率）。
- Schema 驗證：使用 Pydantic 模型定義 API 接口，確保前後端通訊協議一致。

### 7.2 單元測試（Unit Testing）

- 邏輯抽離：將模擬邏輯（PyAEDT）與伺服器邏輯（FastAPI）分離，方便針對單一功能編寫測試腳本。
- pyaedt / pyedb 腳本以參數方式傳遞，並另外撰寫測試腳本。

## 8. 數據視覺化與工程報表策略

將模擬結果轉化為可決策的資訊，讓使用者不只看到數值，也能快速理解差異與風險。

### 8.1 交互式視圖（Interactive View）

- 多維度對比：讓使用者能在同一張 Plotly 圖表中對比不同版本（Versions）或不同 Case 的 S 參數。
- 過渡帶分析：自動標註並判定 Pass / Fail（如 Mask 檢查）。

### 8.2 自動化報表產出

- PDF / HTML 導出：結合 Jinja2 模板與 Matplotlib，一鍵生成正式的工程報表。
- 數位孿生預覽：在介面上直接渲染 EDA 工具生成的 3D 結構圖，方便確認物理模型的正確性。

## 9. 說明書與知識傳承

- API 文件：自動根據 Docstring 生成（如 Sphinx），方便後續維護與二次開發。
- 使用手冊：專為 UI 使用者撰寫的圖文點擊手冊與功能說明。
- 範例庫與教材：提供標準的範例專案，作為 AI 學習或新人培訓的基準。

## 10. 持續更新與迭代

一個成功的自動化系統並非一蹴可幾，而是需要透過不斷的迭代與優化來持續進化。這個過程仰賴於結構化的回饋循環與敏捷的開發文化。

### 10.1 敏捷開發與回饋循環

- 小步快跑：採用敏捷開發（Agile）原則，將大型功能拆解為小而可交付的任務。每次迭代（Sprint）都專注於解決一到兩個核心問題或增加一個實用功能。
- 用戶回饋機制：在系統中建立一個簡單的意見回饋或問題回報按鈕，讓終端使用者能輕鬆提交遇到的問題、操作不便之處或新功能建議。
- 定期審查會議：定期（例如每兩週）與核心使用者開會，展示最新的開發進度，並收集他們對系統的直接回饋。

### 10.2 功能需求的優先級排序

- 影響力 vs. 成本分析：以簡單的四象限矩陣（高影響力 / 低成本、高影響力 / 高成本等）來評估每個待辦事項，優先處理能以最小成本帶來最大效益的功能。
- 錯誤修復優先：系統穩定性是使用者信任的基礎。通常，修復已知錯誤（Bug）應比開發新功能更高優先級。

## 結語

從單純的腳本撰寫到開發大型模擬自動化系統，關鍵在於把架構設計與開發維運拆分治理。以清晰架構支撐功能擴展，以穩定流程支撐長期演進，才能建立真正可持續的工程資產流水線。
