## 1. 核心架構與基本原則

* **技術棧定義**：後端主體基於 **Python 3.10** 開發，前端採用 **React + TypeScript** 現代化框架構建高效能單頁應用 (SPA)。
* **環境管理策略 (uv-based)**：全系統採用 **uv** 作為統一的環境管理與依賴解析工具。
    * **核心服務層**：`backend/core/` 下的所有微服務共享同一個由 `uv` 管理的虛擬環境。
    * **模擬執行層**：每個 `backend/services/` 下的模擬插件維持獨立環境，並透過 `uv run` 或 `uv venv` 實現極速構建與隔離。
    * **一鍵部署機制**：提供 `scripts/setup_env.ps1` 腳本，利用 `uv` 的全域快取（Global Cache）與硬連結（Hard-linking）特性，自動遞迴掃描並建立全系統所有 `.venv` 環境。
* **模擬驅動核心**：深度整合 **PyEDB** 與 **PyAEDT** 開放原始碼套件，實現對 ANSYS 旗艦級模擬器（HFSS, SIwave, Q3D, Icepak 等）的自動化調度與物理場求解。
* **微服務架構 (Microservices)**：全系統採高度解耦設計，並提供整合式啟動腳本實現「一鍵式服務發現與負載初始化」。
* **輕量級數據持久化**：選用 **SQLite 嵌入式資料庫** 集中管理使用者權限、系統配置及 Job 歷史狀態，確保部署靈活性與查詢效能。
* **高性能傳輸策略**：遵循「路徑引用優先」原則，跨服務處理大型模擬專案時僅傳遞專屬路徑 (URI)，嚴禁在 API 或訊息載荷中周轉傳遞二進位檔案。
* **UI 與數據合約一致性**：強制執行統一的 HTML Template 與標準化 JSON Schema。
* **Schema-Driven UI**：前端採動態渲染機制。各模擬服務透過 API 提供其專屬的預算規範 (`schema.json`)，前端根據此 Schema 自動生成對應的輸入表單，實現「新增服務、無須更動前端代碼」的高擴充性。
* **管理者掛載與啟動流程（`backend/services`）**：管理者僅需將符合合約之 **Simulation Service** 目錄放入 **`backend/services/<service_id>/`**（目錄名稱即建議之 `service_id`），**無須**手動修改 Portal 原始碼或另行建立註冊檔。具 **Administrator** 身分之使用者登入**管理後台**後，系統須透過**目錄掃描**自動列出該服務（含「未啟用／待安裝」等狀態）。管理者於後台對該服務執行「**啟動**」時，系統須**自動**完成該服務之 **`.venv` 建立與依賴安裝**（與 **`uv venv`／`uv sync`** 或專案封裝之等價流程一致；可內部調用 `scripts/setup_env.ps1` 之邏輯，但**不得**要求管理者先手動於終端機執行腳本）。啟動流程須一併完成與 **Portal** 之資訊連線：**Service Registry** 之註冊或更新、服務清單與 **`schema.json`** 之下發路徑、健康檢查／心跳與狀態回報、以及與 **Job／Scheduler** 可派發對象之關聯。流程完成後，**使用者頁面**（依權限顯示之儀表板入口、動態表單、Job 追蹤等）與**管理者頁面**（服務清單、監控、啟停、熱部署）**皆須**可見該服務；使用者端未獲授權者依權限矩陣顯示唯讀或隱藏，不在本條豁免範圍外之「不可見」。
* **插件式目錄掛載（Simulation Service）**：**Simulation Service**（模擬執行層）須支援「將符合合約之目錄置於 **`backend/services/<service_id>/`** 後，依 **「管理者掛載與啟動流程」** 由後台掃描、啟動並向系統註冊」之擴充模式。**禁止**在核心程式中硬編碼單一服務之**檔案系統路徑**。服務就緒後須向 **Service Registry**（由 Portal 後台維護，見 **Simulation Service 連線位址由 Portal 統一發放與管理**）註冊或更新其可連線端點，供排程與閘道路由使用。
* **插件式目錄掛載（Simulation Portal）**：**Simulation Portal**（門戶／前端與入口整合層）之插件須支援目錄化掛載。將符合 Portal 插件合約的模組目錄（如路由註冊、選單 metadata、動態載入設定）置於約定位置（例如 `frontend/src/plugins/<plugin_id>/` 或專案文件所定義之掃描根目錄），納入主專案建置管線後即可啟用；**不得**依賴開發機絕對路徑。插件程式碼內 **禁止** 寫死各 Simulation Service 之 IP、埠或完整 Base URL（須改由 Portal 下發之服務發現資料驅動）。
* **Simulation Service 連線位址由 Portal 統一發放與管理**：各 Simulation Service 之對外可連線位址（**IP 或 Hostname、埠、Base URL、反向代理路徑前綴** 等）由 **Portal 所屬之服務註冊／管理後台與 Service Registry** 作為 **單一事實來源（SSOT）** 統一登錄、變更、下發與撤銷；**禁止**於前端 SPA 原始碼、Portal 插件、使用者可見之靜態設定、`config.json` 範本或文件範例中寫死個別 Simulation Service 之 IP。使用者瀏覽器預設僅連線至 **Portal／API 閘道或 BFF**；轉發至各 Simulation Service 時，由 **伺服器端** 依 Registry 解析，避免內網 IP 被打進前端建置產物。**Scheduler**、**Job & Queue Service** 或其他後端元件派發或回撥 Simulation Service 時，須讀取 **同一 Registry**，不得各自維護一份硬編碼位址清單。開發環境之 `.env` 僅允許設定 **閘道／Portal 根位址** 等全域項，**不得**作為繞過 Registry、逐服務寫死 IP 之長期手段。
* **可攜性聲明**：「目錄移入即可運作」係指在 **相同作業系統與 Python/Node 主版本**、且管理者已於後台完成「啟動」（含自動 `.venv` 與 Registry 連線）之前提下成立；生產環境仍須遵循容器化或自動化部署流程，並以健康檢查與煙霧測試驗證掛載成功。

## 2. 角色功能需求

### 2.1 使用者 (User)

* **身份認證**：首次進入系統需完成註冊，經管理者核對權限後方可存取。
* **儀表板 (Dashboard)**：登入後顯示個人主頁。系統根據獲授權服務清單動態渲染入口；未獲權限之服務自動進入「唯讀/停用」狀態。
* **任務生命週期管理**：使用者可實時查看 Job 的排隊深度、執行進度與歷史記錄。支援任務取消 (Delete Job) 與一鍵下載封裝後的模擬報告 (Simulation Directory Artifacts)。
* **交易安全防護**：提交模擬請求時啟動 UI 鎖定機制 (Request Locking)，待後端 Job ID 返回並建立工作目錄後方能解除，防止重複提交。

### 2.2 管理者 (Administrator)

* **服務治理**：提供專屬管理後台。管理者將服務目錄放入 **`backend/services/<service_id>/`** 後，後台須**自動掃描**並列出該服務；支援「**啟動**」（觸發 **§1 管理者掛載與啟動流程** 所述之 `.venv` 安裝與 Portal 全量連線）、「**重啟**」、「**下架**」與熱部署相關操作。啟動完成後，該服務須同時出現於**使用者可見之服務清單／儀表板**（依權限）與**管理者監控／服務列表**。
* **服務端點統一維護**：於管理後台維護各 **Simulation Service** 之連線位址（IP/Hostname、埠、Base URL 或閘道路徑；若啟動流程已自動偵測並寫入 Registry，管理者僅需於異常或遷機時覆寫），並寫入由 **Portal** 管理之 **Service Registry**；變更後透過重新載入、快取失效或版本化 API 使全系統（含 Scheduler、閘道、前端服務清單）取得一致之下發結果，**不得**要求工程師改前端或插件原始碼內之硬編碼 IP。
* **全局調度控制**：監控全局任務隊列 (Global Queue)。管理者有權變更 Job 優先權 (Priority Adjustment) 或進行任務插隊。
* **權限矩陣配置**：精細化管理使用者存取路徑，提供一鍵「模擬權限賦予/撤銷」功能。
* **數據分析看板**：整合 Metrics Service 數據，監控系統健康度、算力利用率及使用者資源佔用趨勢。
* **趨勢分析圖表**：提供時間維度的圖表分析，協助評估算力需求與系統負載變化。

## 3. 微服務組件功能定義

### 3.1 User & Auth Service (認證微服務)

* **身份驗證**：連結專屬 SQLite 資料庫，負責使用者註冊、登入憑證驗證與安全 Token (如 JWT) 的發放。
* **權限管理**：管理使用者權限矩陣，動態輸出權限清單以控制前端功能按鈕的啟用狀態。

### 3.2 Notification & Logging Service (通知與日誌微服務)

* **即時通訊**：負責所有微服務與前端介面之間的「即時資訊同步」。當 Job 狀態異動時，即時推送到相關頁面。
* **日誌流轉**：接收來自所有服務的運行日誌 (Log)，同步寫入系統日誌並實時滾動呈現於前端控制台。

### 3.3 Metrics & Analytics Service (統計數據微服務)

* **數據採集**：紀錄 Job 完整生命週期 (建立、排隊、執行、完工/報錯)，精確統計耗時、成功率及資源消耗。
* **數據聚合**：負責數據清洗與多匯總，產出統計報表 (如：日/周/月使用量趨勢、各類模擬比例)。
* **分析維度**：
    * **服務維度**：統計各服務請求熱度、平均執行效率、資源利用率與故障分布。
    * **使用者維度**：統計個別使用者累計運行次數、總消耗算力與服務分佈。

### 3.4 Job & Queue Service (任務與佇列微服務)

* **工作目錄管理**：使用者提交請求時，在 `jobs/Job_<timestamp>/` 建立專屬目錄，存放上傳檔案並生成 `config.json`。
* **任務狀態登記**：管理全系統提交任務的登記狀態，並在異動時向全系統同步資訊。

### 3.5 Scheduler (排程器微服務)

* **資源感知**：實時監測 Windows 系統資源與 Ansys License 餘量。啟動時自動檢查 License Server，必要時重啟 `lmgrd`。
* **排程演算法**：預設 FIFO 機制，具備 Backfilling 回填能力，在資源不足時自動調度小型 Job 優先執行以提高吞吐量。
* **進程監管 (Watchdog)**：
    * **生命週期管理**：透過 REST API 派發任務，並持續追蹤 Work 進程及其產生的子進程。
    * **超時清理**：當 Job 超過 `max_runtime_minutes` 時執行 Hard Timeout 清理，並回收系統資源與 License。
    * **異常感知**：定期進行 Heartbeat 監測，若 Simulation Service 中斷則立即啟動清理流程。

### 3.6 Simulation Service (模擬執行微服務)

* **插件式交付**：每一服務為自包含目錄；管理者將目錄置於 **`backend/services/<service_id>/`** 並於後台「啟動」後，系統須自動完成 `.venv` 與依賴同步（見 **§1 管理者掛載與啟動流程**），無須修改核心微服務原始碼。
* **連線位址與 Registry**：服務啟動後須向 **Portal 統一管理之 Service Registry** 註冊或更新其可連線端點（或由管理者於後台綁定後由服務確認）；**不得**假設呼叫端（含前端）已於程式內寫死本服務 IP。對外實際 IP／埠以 Registry 為準，遷機或改埠僅需更新 Registry 與健康檢查。
* **高擴充架構**：模擬工序 (HFSS, SIwave, Q3D 等) 封裝為獨立套件，支援動態掛載 (Dynamic Loading)。
* **自描述能力**：包含 `schema.json` 描述輸入參數與驗證規則，前端據此動態渲染 UI。
* **環境隔離策略**：每個服務透過 **uv** 構建專屬環境，並以 `uv run` 啟動獨立子進程進行模擬。
* **數據與日誌追溯**：
    * **數據獨立性**：嚴格落實「按路徑存取」，所有中間檔案與結果均落地於該 Job 目錄。
    * **日誌持久化**：實時捕獲 stdout/stderr 轉發至前端，同時將 PyAEDT/PyEDB 日誌寫入 `output/logs/pyaedt.log`。
* **核心邏輯解耦**：PyAEDT/PyEDB 核心腳本必須與服務框架代碼分離，確保邏輯清晰且易於調測。
* **自動清理及進度顯示**：完工後自動封裝 `.aedtz` 並移除實體專案檔；處理大體積數據時提供前端進度條。

### 3.7 Utility Services (工具型微服務)

* **數據萃取**：負責 `.aedb` 或 CAD 檔案的輕量級解析 (如：讀取 Nets, Pins, Components)。
* **高性能響應**：直接透過 **PyEDB** 讀取底層數據，無須啟動 AEDT Desktop，確保 API 響應在秒級以內。
* **無狀態原則**：僅接受檔案路徑並回傳 JSON 數據，不存儲 Job 狀態。
* **跨服務共享**：作為系統基層組件，可同時供給前端 UI 預覽與多個 Simulation Services 進行前置參數驗證。

### 3.8 Simulation Portal 插件化門戶（與服務連結）

* **門戶定位**：Simulation Portal 指使用者-facing 之 SPA 與其插件化擴充層，負責儀表板、動態表單、Job 追蹤與管理後台介面；與 **Simulation Service** 透過統一 API 閘道與服務清單解耦。
* **目錄即插件**：新增門戶功能時，以約定目錄結構提交（路由、國際化資源、選單與服務對應表等），納入主專案建置管線後即可啟用；與 **§1 插件式目錄掛載（Simulation Portal）** 原則一致。
* **服務位址統一發放**：Portal（含其後台與 **Service Registry**）為各 **Simulation Service** 連線位址之 **唯一管理與下發來源**；前端與插件僅透過 **服務發現／服務清單 API** 取得邏輯 `service_id` 與由閘道代理之可呼叫路徑，**禁止**在 UI 層寫死 IP 或埠。後端回傳之服務 metadata（`service_id`、顯示名稱、`schema.json` 位址或內嵌、可選之閘道相對路徑等）驅動表單與入口；服務經 **`backend/services` 掛載並於後台啟動** 後，須自動納入清單並使**使用者頁**與**管理者頁**同步可見（使用者端仍受權限矩陣約束）。服務目錄增刪或遷機後，經 Registry 更新與快取失效即可全站一致，無須改前端硬編碼。

### 附錄3: 系統通訊協議概覽

* **Command (命令)**：採 **RESTful API** (HTTP JSON)，負責 Job 提交、狀態查詢、權限異動等請求。
* **Real-time Log (即時日誌)**：採 **WebSocket (WS)** 或 **SSE**，將 stdout/stderr 持續推送至前端。
* **Environment (環境變數)**：全系統採 `.env` 進行密鑰 (如 JWT Secret)、**Portal／API 閘道根位址** 與核心路徑 (Portal Root) 等全域項之配置管理；**各 Simulation Service 之 IP 不得**作為長期設定寫入 `.env` 取代 **Service Registry**（見 **§1 Simulation Service 連線位址由 Portal 統一發放與管理**）。

### 附錄4: 技術框架與 Python 模組建議

#### 4.1 後端核心框架
* **FastAPI**: 用於建構高效能的 REST API，支援非同步 (asyncio) 運作。
* **Uvicorn / Gunicorn**: 作為 ASGI 伺服器，負責運行 FastAPI 應用程式。
* **Pydantic**: 用於數據格式驗證與標準化 `config.json` 的模型定義。

#### 4.2 模擬與工程自動化
* **ansys-aedt-core (PyAEDT)**: 負責調用 AEDT (HFSS, Maxwell, Q3D 等) 進行腳本化建模與求解。
* **pyedb**: 專門用於處理 PCB 與 封裝數據 (EDB 格式) 的自動化修改與萃取。

#### 4.3 即時通訊與併發處理
* **Socket.io / FastAPI-SocketIO**: 實現 WebSocket 通訊，用於將模擬日誌即時推送到前端。
* **Loguru**: 用於結構化日誌記錄，支援非同步寫入。
* **Subprocess API**: 用於 Scheduler 調用 `uv run` 啟動模擬子進程，並進行監控清。

#### 4.4 數據持久化與工具
* **SQLAlchemy / SQLModel**: 作為 SQLite 的 ORM，簡化資料庫操作。
* **Alembic**: 用於資料庫遷移管理 (Database Migration)。
* **Python-dotenv**: 用於讀取 `.env` 環境變數。

#### 4.5 前端開發棧
* **React**: 主要 UI 框架。
* **TypeScript**: 確保前端代碼的型別安全與開發效率。
* **Tailwind CSS / Ant Design**: 用於快速建構專業且一致的工程介面。
* **Axios**: 用於與後端微服務進行 HTTP 通訊。
